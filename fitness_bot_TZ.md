# Техническое задание: Telegram-бот управления телом

**Версия:** 1.0  
**Дата старта:** 1 июня  
**Цель:** персональный AI-трекер веса, питания и тренировок с циклическим дефицитом калорий  

---

## 1. Цели и контекст проекта

### 1.1 Суть задачи

Создать Telegram-бота, который работает как персональный тренер по данным. Бот знает конкретный план пользователя, применяет его каждый день и следит за прогрессом на пути к цели −20 кг (старт: 96 кг → цель: ~76–78 кг) за 24 недели.

Это **не просто трекер**. Бот знает:
- какой сегодня день недели → значит знает целевые калории на сегодня
- тренировочный это день или отдых → меняет советы по питанию
- на каком цикле сейчас находится пользователь → корректирует поведение
- были ли признаки адаптации → предупреждает и даёт рекомендации

### 1.2 Профиль пользователя (зашит в систему)

| Параметр | Значение |
|---|---|
| Стартовый вес | 96 кг |
| Цель | ~76–78 кг |
| Срок | 24 недели (старт 1 июня) |
| Тренировки | Пн / Ср / Пт — full body |
| Окно питания | 16/8 (11:00–19:00) |
| NEAT | низкий |
| Белок | 160–190 г ежедневно (не снижать) |
| Шаги | минимум 6000–8000 в день |

## 2. Калорийный цикл (бизнес-правила ядра)

### 2.1 Недельный цикл калорий

| День | Тип | Целевые ккал | Акцент |
|---|---|---|---|
| Пн | Тренировка | 1900 | больше углеводов, питание вокруг тренировки |
| Вт | Отдых | 1600 | низкие углеводы, высокий белок |
| Ср | Тренировка | 1900 | больше углеводов |
| Чт | Отдых | 1600 | низкие углеводы, высокий белок |
| Пт | Тренировка | 1900 | больше углеводов |
| Сб | Отдых | 1600 | низкие углеводы, высокий белок |
| Вс | Лёгкая активность | 1700 | умеренно, чтобы не срываться в понедельник |

**Среднее за неделю:** ~1743 ккал/день  
**Итого за неделю:** 12 200 ккал

### 2.2 Схема питания внутри дня

**Тренировочный день (1900 ккал), окно 11:00–19:00:**

| Время | Приём | Содержание |
|---|---|---|
| 11:00 | Основной | Белок + углеводы |
| 15:00 | Перед залом | Лёгкие углеводы + белок |
| 18:30 | После зала | Белок + углеводы + жиры |

**День отдыха (1600 ккал), окно 11:00–19:00:**

| Время | Приём | Содержание |
|---|---|---|
| 11:00 | Основной | Белок + жиры (плотный) |
| 15:00 | Промежуточный | Белок |
| 18:30 | Завершающий | Белок + немного углеводов |

### 2.3 Система 6 циклов (24 недели)

| Цикл | Недели | Ккал (среднее) | Ожидаемый результат |
|---|---|---|---|
| Цикл 1 | 1–4 | 1750–1850 | −4 до −6 кг |
| Цикл 2 | 5–8 | 1700–1750 | −3 до −4.5 кг |
| Цикл 3 | 9–12 | 1650–1700 | −3 до −4 кг |
| Diet Break | 13 | 2200–2400 | восстановление гормонов |
| Цикл 4 | 14–17 | 1700–1800 | −3 до −4 кг |
| Цикл 5 | 18–21 | 1650–1750 | −2.5 до −4 кг |
| Цикл 6 | 22–24 | 1550–1700 | −2 до −4 кг |

**Diet Break — это не срыв.** Бот должен объяснять это явно, когда наступает 13-я неделя.

### 2.4 Ожидаемый прогресс по весу (коридоры)

| Неделя | Вес (коридор) |
|---|---|
| 1–4 | 96 → 90–92 кг |
| 5–8 | 90 → 86–88 кг |
| 9–12 | 86 → 82–84 кг |
| 13 (diet break) | 83–85 кг (возможен +1–2 кг воды — норма) |
| 14–17 | 85 → 80–82 кг |
| 18–21 | 81 → 77–80 кг |
| 22–24 | 78 → 76–78 кг |

### 2.5 Признаки адаптации (бот должен отслеживать и сигнализировать)

Бот сигнализирует о возможной адаптации, если обнаруживает 2+ признака одновременно:

1. Вес стоит 10–14 дней при соблюдении калорий
2. Пользователь не вносит данные по шагам (снижение NEAT)
3. В тренировках снижение нагрузки 2 недели подряд
4. Пользователь сообщает о постоянном голоде
5. Ухудшение качества сна (по сообщениям)

**При адаптации бот рекомендует (по порядку):**
1. Проверить реальность учёта (скрытые перекусы?)
2. Добавить +2000–3000 шагов в день
3. Если 2+ недели стоит — diet break (5–7 дней на 2200–2400 ккал)
4. Циклирование углеводов (тренировки ↑, отдых ↓)
5. Только потом — снижение калорий на 100–150

## 3. Стек технологий

### 3.1 Выбранный стек

| Компонент | Технология | Обоснование |
|---|---|---|
| Telegram Bot | grammY (Node.js + TypeScript) | TypeScript-first, лучший middleware |
| Хостинг | Railway | free tier не засыпает (500 ч/мес), нативный cron |
| База данных | Supabase (PostgreSQL) | pgvector, Auth, Storage, Cron — всё в одном |
| AI основной | Groq API (Llama 3.3 70B) | бесплатно, ~0.3 сек ответ |
| AI fallback | OpenRouter (DeepSeek) | запасной, если Groq недоступен |
| База продуктов | Open Food Facts + USDA FoodData | Open Food Facts — RU продукты, USDA — fallback |
| Векторная память | Supabase pgvector | без отдельного сервиса |
| Cron / уведомления | Railway Cron + Supabase Scheduled | напоминания, ежедневный отчёт |
| Аналитика | Telegram WebApp (Chart.js) | графики прогресса прямо в боте |

## 4. Структура базы данных

### 4.1 Таблица `users`

```sql
CREATE TABLE users (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tg_id        BIGINT UNIQUE NOT NULL,
  username     TEXT,
  start_weight DECIMAL(5,2) DEFAULT 96.0,
  goal_weight  DECIMAL(5,2) DEFAULT 77.0,
  start_date   DATE DEFAULT '2024-06-01',
  current_cycle INT DEFAULT 1,
  created_at   TIMESTAMPTZ DEFAULT NOW()
);
```

### 4.2 Таблица `weights`

```sql
CREATE TABLE weights (
  id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id   UUID REFERENCES users(id),
  date      DATE NOT NULL,
  weight    DECIMAL(5,2) NOT NULL,
  logged_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id, date)
);
```

### 4.3 Таблица `food_logs`

```sql
CREATE TABLE food_logs (
  id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id   UUID REFERENCES users(id),
  date      DATE NOT NULL,
  meal_time TEXT,           -- '11:00', '15:00', '18:30'
  food_name TEXT NOT NULL,
  grams     DECIMAL(7,2),
  calories  DECIMAL(7,2),
  protein   DECIMAL(7,2),
  fat       DECIMAL(7,2),
  carbs     DECIMAL(7,2),
  source    TEXT,           -- 'open_food_facts', 'usda', 'manual'
  logged_at TIMESTAMPTZ DEFAULT NOW()
);
```

### 4.4 Таблица `frequent_foods` (избранные/часто используемые)

```sql
CREATE TABLE frequent_foods (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id    UUID REFERENCES users(id),
  food_name  TEXT NOT NULL,
  grams      DECIMAL(7,2),
  calories   DECIMAL(7,2),
  protein    DECIMAL(7,2),
  fat        DECIMAL(7,2),
  carbs      DECIMAL(7,2),
  use_count  INT DEFAULT 1,
  last_used  DATE,
  UNIQUE(user_id, food_name)
);
```

Логика: если продукт добавлен 3+ раз — попадает в "частые". При следующем вводе похожего продукта — бот предлагает его первым через inline-клавиатуру.

### 4.5 Таблица `workouts`

```sql
CREATE TABLE workouts (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id    UUID REFERENCES users(id),
  date       DATE NOT NULL,
  type       TEXT DEFAULT 'full_body',  -- 'full_body', 'rest', 'light'
  exercises  JSONB,   -- [{name, sets, reps, weight_kg}]
  notes      TEXT,
  duration_min INT,
  logged_at  TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id, date)
);
```

### 4.6 Таблица `daily_summary`

```sql
CREATE TABLE daily_summary (
  id                 UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id            UUID REFERENCES users(id),
  date               DATE NOT NULL,
  day_type           TEXT,          -- 'workout', 'rest', 'light'
  target_calories    INT,           -- по расписанию (1600 / 1700 / 1900)
  total_calories     DECIMAL(7,2),
  total_protein      DECIMAL(7,2),
  total_fat          DECIMAL(7,2),
  total_carbs        DECIMAL(7,2),
  steps              INT,           -- вносится вручную с часов
  steps_unavailable  BOOLEAN DEFAULT FALSE,  -- часы не носил
  weight             DECIMAL(5,2),
  status             TEXT,          -- 'deficit', 'on_target', 'over'
  UNIQUE(user_id, date)
);
```

### 4.7 Таблица `measurements`

```sql
CREATE TABLE measurements (
  id      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id),
  date    DATE NOT NULL,
  waist   DECIMAL(5,2),
  neck    DECIMAL(5,2),
  chest   DECIMAL(5,2),
  hips    DECIMAL(5,2),
  UNIQUE(user_id, date)
);
```

### 4.8 Таблица `memory_vectors` (RAG)

```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE memory_vectors (
  id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id   UUID REFERENCES users(id),
  text      TEXT NOT NULL,
  embedding VECTOR(1536),
  type      TEXT,   -- 'food_preference', 'progress_note', 'ai_insight'
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX ON memory_vectors USING ivfflat (embedding vector_cosine_ops);
```

## 6. Команды и inline-клавиатуры

### 6.1 Стартовое меню `/start`

Бот отвечает приветствием и показывает главную inline-клавиатуру:

```
┌─────────────────────────────────────┐
│  Привет! Сегодня [ДЕНЬ НЕДЕЛИ]      │
│  Тип дня: 🏋️ Тренировка / 😴 Отдых │
│  Цель на сегодня: 1900 / 1600 ккал  │
└─────────────────────────────────────┘

[⚖️ Внести вес]    [🍽️ Питание]
[🏋️ Тренировка]   [📊 Итог дня]
[🚶 Шаги]          [📈 Прогресс]
[🤖 Спросить AI]
```

> Шаги вносятся вручную с часов Traverse Alpha — никакой API-интеграции нет и не нужно. Просто число.

### 6.2 Команда `⚖️ Внести вес`

Пользователь нажимает кнопку. Бот спрашивает:

```
Введи вес утром, например: 95.4
```

После ввода числа:

```
✅ Записал: 95.4 кг

📉 Тренд за 7 дней: −0.6 кг
📍 Неделя 3, Цикл 1
🎯 Цель на цикл: 90–92 кг (осталось ~4.2 кг)

[➕ Добавить ещё] [🏠 Меню]
```

Если вес вырос после diet break — бот объясняет:

```
⚠️ +1.3 кг — это вода и гликоген от diet break.
Жир не растёт. Продолжай план, к концу недели уйдёт.
```

### 6.3 Команда `🍽️ Питание`

Подменю:

```
[➕ Добавить еду]     [⭐ Частые продукты]
[📋 Сегодня]          [🔍 Поиск продукта]
[🏠 Меню]
```

**Добавление еды — текстовый ввод:**

Пользователь пишет: `курица 200г рис 100г`

Бот парсит регулярными выражениями:

```
Распознал:
• Куриная грудка 200г — 220 ккал, Б 42г, Ж 4г, У 0г
• Рис варёный 100г — 130 ккал, Б 3г, Ж 0г, У 28г

Итого добавляю: 350 ккал

[✅ Подтвердить]  [✏️ Изменить]  [❌ Отмена]
```

После подтверждения — бот показывает итог дня:

```
📊 Сегодня (тренировочный день, цель 1900 ккал):
Ккал: 1340 / 1900 — осталось 560
Белок: 98г / 165г
Жиры: 32г
Углеводы: 88г

Статус: в дефиците ✅
```

**Частые продукты (inline-клавиатура):**

Бот показывает топ-8 продуктов, которые пользователь добавлял чаще всего:

```
⭐ Твои частые продукты:

[🍗 Куриная грудка]  [🥚 Яйца]
[🥣 Творог 5%]       [🐟 Лосось]
[🍚 Рис варёный]     [🥦 Брокколи]
[🥜 Арахисовая паста][🧀 Сыр]

[+ Другой продукт]   [🏠 Меню]
```

При нажатии на продукт — бот спрашивает граммы:

```
Куриная грудка — сколько грамм?

[100г] [150г] [200г] [250г] [Другое]
```

**Поиск продукта:**

Пользователь пишет название. Бот ищет в Open Food Facts → USDA. Показывает до 5 результатов:

```
Нашёл по запросу "творог":

[Творог 5% Простоквашино]
[Творог 0% обезжиренный]
[Творог зернёный 5%]
[Творог домашний (USDA)]
```

### 6.4 Команда `🏋️ Тренировка`

```
Сегодня тренировочный день (Full Body).
Хочешь записать тренировку?

[✅ Записать]  [⏭️ Пропустить]  [📝 Только заметку]
```

При записи — бот предлагает быстрый ввод:

```
Напиши упражнения, например:
"Присед 5×5 80кг, Жим 4×8 60кг, Тяга 3×10 70кг"

Или нажми "Пропустить детали" — запишу просто факт тренировки.

[Пропустить детали]  [❌ Отмена]
```

После записи:

```
✅ Тренировка записана!

Сегодня тренировочный день → цель 1900 ккал.
Не забудь поесть углеводы после зала (до 19:00).

[🍽️ Питание]  [🏠 Меню]
```

### 6.5 Команда `🚶 Шаги`

Шаги вводятся вручную — пользователь смотрит на часы Traverse Alpha и вписывает число.

Бот спрашивает:

```
Сколько шагов сегодня?
Посмотри на часы и введи число, например: 7450
```

После ввода числа:

```
✅ Записал: 7 450 шагов

Цель: 6 000 шагов — выполнено ✅

[🏠 Меню]
```

Если шагов меньше 6000:

```
✅ Записал: 4 200 шагов

⚠️ До цели: ещё 1 800 шагов.
NEAT напрямую влияет на дефицит — постарайся добрать к вечеру.

[🏠 Меню]
```

Если шагов меньше 4000:

```
✅ Записал: 3 100 шагов

🔴 Очень мало. Без шагов адаптация "съедает" дефицит.
Даже 20 минут прогулки = +2000 шагов.

[🏠 Меню]
```

**Когда вносить шаги:** в любое время дня, но лучше вечером — чтобы была итоговая цифра за день. Бот напоминает внести шаги в 20:00 вместе с итогом дня если они не внесены.

**Напоминание в 20:00 если шаги не внесены:**

```
🚶 Шаги за сегодня не записаны.
Посмотри на часы и внеси — это секунда.

[🚶 Внести шаги]  [Часы не носил]
```

Если "Часы не носил" — день помечается `steps_unavailable = true`, бот не считает этот день при анализе NEAT.

### 6.6 Команда `📊 Итог дня`

Вызывается вручную или автоматически в 20:00:

```
📊 Итог дня — Пн, 3 июня (неделя 1, цикл 1)
Тип дня: 🏋️ Тренировка

⚖️ Вес: 95.4 кг
🔥 Ккал: 1820 / 1900 — дефицит ✅
🥩 Белок: 172г / 165г — выполнено ✅
🧈 Жиры: 58г
🍞 Углеводы: 142г
🚶 Шаги: 7200 / 6000 — выполнено ✅
🏋️ Тренировка: проведена ✅

Статус дня: ОТЛИЧНЫЙ ⭐

Недельный дефицит (5 из 7 дней):
Ср. дефицит: ~320 ккал/день → −0.32 кг жира за неделю
```

### 6.7 Команда `📈 Прогресс`

Открывает Telegram WebApp с графиками:

```
[📈 Открыть графики]

Или текстовая версия:
• Старт: 96 кг (1 июня)
• Сейчас: 95.4 кг
• Прошло: 3 дня
• Потеряно: −0.6 кг
• Цикл 1, неделя 1
• До цели: 18.4 кг
```

### 6.8 Команда `🤖 Спросить AI`

Пользователь пишет любой вопрос на русском языке. AI получает контекст:

- текущий вес, цикл, неделя
- калории за сегодня и за неделю
- тип сегодняшнего дня
- история за последние 7 дней
- частые продукты пользователя

Примеры вопросов, на которые бот знает ответ:

- "что мне съесть сегодня?"
- "я не потерял вес за 10 дней — что делать?"
- "можно ли сделать читмил?"
- "как накормить мышцы без перебора?"

## 8. NLP парсер питания

### 8.1 Форматы, которые бот понимает

```
курица 200г рис 100г
творог 5% 150 грамм
2 яйца
яблоко среднее
лосось на пару 180г огурец
```

### 8.2 Алгоритм парсинга

1. Регулярное выражение вычленяет пары `(продукт, количество)`:
   - `(\d+)\s*(г|гр|грамм|г\.)?` — граммы
   - `(\d+)\s*(шт|штук|яйца|яйцо)` — штуки (конвертируются в граммы)
   - Без цифр — условная порция (запрашивается уточнение)

2. Нормализация названия продукта (lowercase, убрать лишние слова)

3. Поиск в `frequent_foods` пользователя (если совпадение ≥ 80% — предлагается сразу)

4. Если не нашли в частых → Open Food Facts API

5. Если не нашли в Open Food Facts → USDA FoodData Central API

6. Если не нашли нигде → AI estimation (Groq) с пометкой "примерно"

7. Пользователю показывается результат с кнопками: `[✅ Подтвердить]` / `[✏️ Изменить граммы]` / `[❌ Другой продукт]`

### 8.3 Учёт частых продуктов

- При каждом подтверждении продукта: `use_count++`, `last_used = today`
- Если `use_count >= 3` — продукт появляется в inline-клавиатуре "частые"
- В кнопке "частые" — топ 8 по `use_count` за последние 30 дней
- Предложение граммов: последнее использованное количество как кнопка по умолчанию

## 9. Недельный дефицит и аналитика

### 9.1 Расчёт дефицита

Бот ежедневно считает:

```
actual_deficit = target_calories_for_day - actual_calories
weekly_deficit = sum(daily_deficits) за текущую неделю (пн–вс)
estimated_fat_loss = weekly_deficit / 7700  # кг жира
```

### 9.2 Статусы дня

| Статус | Условие | Что показывает бот |
|---|---|---|
| ОТЛИЧНЫЙ ✅ | ккал в пределах ±100, белок ≥ 155г, шаги ≥ 6000 | хвалит |
| НОРМА ✅ | ккал в пределах ±150 | нейтрально |
| ПЕРЕБОР ⚠️ | ккал > цель + 200 | предупреждает, не осуждает |
| НЕДОЕЛ ⚠️ | ккал < цель − 300 | объясняет риск адаптации |
| КРИТИЧНО 🔴 | белок < 140г | предупреждает — это красная линия |

## 13. Важные правила работы бота

1. **Белок — красная линия.** Если белок < 140г — бот всегда предупреждает, независимо от остального.

2. **Не компенсировать дни.** Если пользователь говорит "вчера ел мало — сегодня можно больше?" — бот объясняет, почему это плохо.

3. **Diet break — не срыв.** Бот никогда не говорит "ты отдыхаешь от диеты". Он говорит "это часть плана — восстановление гормонов".

4. **Плато ≠ провал.** При вопросе о стоящем весе — бот первым делом проверяет реальность учёта, потом — шаги, потом — recommends diet break.

5. **Никогда не рекомендовать < 1500 ккал** без явного запроса пользователя и предупреждения о рисках.

6. **Шаги — необходимый минимум.** 6000 шагов всегда в контексте напоминаний, без них "адаптация съест дефицит".

7. **Тон бота:** дружелюбный, конкретный, без нотаций. Никогда не говорит "ты нарушил диету". Говорит "сегодня перебор 300 ккал — учти завтра".

## 15. Интеграция с Suunto — автоматический учёт шагов

### 15.1 Реальная ситуация с Suunto API

Suunto Cloud API умеет отдавать шаги и ежедневную активность, но есть ограничение: <br>
**Suunto официально не предоставляет API для личного использования** — только для компаний через партнёрскую программу (срок рассмотрения до 2 недель, нужно подписывать соглашение).

Доступные варианты — от лучшего к запасному:

| Вариант | Сложность | Надёжность | Стоимость |
|---|---|---|---|
| **А. Suunto Cloud API** (через партнёрку) | высокая | высокая | бесплатно |
| **Б. Через Strava** (Suunto → Strava → API) | средняя | высокая | бесплатно |
| **В. Ручной ввод с кнопками** | минимальная | 100% | бесплатно |

Для личного бота рекомендуется: **Б (Strava) как основной + В (ручной) как fallback**.

---

### 15.2 Вариант А — Suunto Cloud API напрямую

Если всё же подаётся заявка в партнёрскую программу (Suunto иногда принимает и индивидуальных разработчиков при наличии внятного описания проекта).

**Как работает:**

Suunto поддерживает вебхук типа `SUUNTO_247_ACTIVITY_CREATED` — срабатывает когда пользователь синхронизирует часы. В теле вебхука приходят сэмплы с полями `StepCount`, `HR`, `EnergyConsumption` с временными метками.

Если хочется получать шаги ежедневно — нужно делать принудительный запрос к API раз в день. Часы обычно синхронизируются с приложением Suunto в момент окончания тренировки, и именно тогда становятся доступны данные по шагам за день.

**Эндпоинты:**
```
GET https://cloudapi.suunto.com/v2/activities/
  ?since={timestamp_начала_дня}
  &limit=1

→ возвращает dailySteps, calories, activeTime
```

**OAuth flow:**
```
1. Пользователь открывает ссылку авторизации Suunto
2. Логинится своим

- Вес вносится **7 из 7 дней** в неделю (100%)
- Питание фиксируется **100% дней**, все приёмы пищи внесены
- AI запросы обрабатываются за < 3 сек (Groq)
- Бот не засыпает (Railway uptime ≥ 99%)
- Пользователь видит тренд веса как скользящее среднее 7 дней (не паникует от суточных колебаний)

## 16. Система "паники" — принудительный контроль данных

Это ключевая система бота. Без данных — нет аналитики, нет тренда, нет смысла. Бот не просит, а **добивается** внесения каждого приёма пищи и ежедневного веса.

### 16.1 Логика приёмов пищи

Каждый день делится на 3 обязательных приёма пищи с временными окнами:

| Приём | Окно ввода | Дедлайн напоминания |
|---|---|---|
| Приём 1 (11:00) | 10:30–13:30 | 13:00 |
| Приём 2 (15:00) | 14:00–17:00 | 16:30 |
| Приём 3 (18:30) | 17:30–20:00 | 19:30 |

### 16.2 Алгоритм напоминаний по весу

**Шаг 1 — Утреннее напоминание (08:00):**
```
🌅 Доброе утро! Время взвеситься.
Натощак, до еды, без одежды.

[⚖️ Внести вес]
```

**Шаг 2 — Повтор если не внесён (09:30):**
```

Это занимает 10 секунд — данные нужны для тренда.

[⚖️ Внести вес]
```

**Шаг 3 — Жёсткое напоминание (11:00), если всё ещё нет:**
```
🚨 ВНИМАНИЕ!

Вес не записан уже 2,5 часа.
Без ежедневного веса тренд неточный — ты не увидишь реального прогресса.

Запишь прямо сейчас, даже если уже поел — запишем с пометкой "не натощак".

[➕ Записать приём 3 (приблизительно)]  [День питания закрыт]
```

### 16.4 Эскалация при систематических пропусках

Бот отслеживает серии пропусков и меняет тон:

**2 дня подряд без веса:**
```
⚠️ Уже 2 дня без записи веса.

Тренд теряет точность. Напомни себе: весы → телефон → бот.
Это 15 секунд. Каждый день.

[⚖️ Внести вес сейчас]
```

**3 дня подряд без веса:**
```
🚨 3 дня без записи веса подряд!

Ты теряешь данные о своём прогрессе.
Без ежедневного взвешивания бот не может:
— показать реальный тренд
— предупредить об адаптации
— подтвердить что план работает

Это НЕ опционально. Взвешивайся каждое утро.

[⚖️ Внести вес]
```

**3+ дня подряд с неполным питанием:**
```
🔴 3 дня подряд питание записано не полностью.

Это разрушает точность недельного дефицита.
Бот не может гарантировать что ты движешься к цели.

Варианты:
1. Записывай всё — это занимает < 2 минут в день
2. Хотя бы приблизительно — лучше "курица примерно 200г" чем ничего

[🤖 Спросить AI как упростить учёт]
```

**Неделя закрыта с пропусками (воскресенье):**

Если за неделю пропущено 2+ дней веса или 3+ приёма:
```
📋 ИТОГИ НЕДЕЛИ [N] — ДАННЫЕ НЕПОЛНЫЕ

⚠️ Пропущено дней веса: [X]/7
⚠️ Пропущено приёмов пищи: [Y]/21

Из-за пропусков недельный дефицит посчитан приблизительно.
Следующая неделя: цель 7/7 дней веса, 21/21 приём пищи.

Точные данные = точный результат.
```

### 16.5 Таблица `data_compliance` — учёт дисциплины

```sql
CREATE TABLE data_compliance (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID REFERENCES users(id),
  date            DATE NOT NULL,
  weight_logged   BOOLEAN DEFAULT FALSE,
  weight_fasted   BOOLEAN DEFAULT TRUE,
  meal1_logged    BOOLEAN DEFAULT FALSE,
  meal1_skipped   BOOLEAN DEFAULT FALSE,
  meal2_logged    BOOLEAN DEFAULT FALSE,
  meal2_skipped   BOOLEAN DEFAULT FALSE,
  meal3_logged    BOOLEAN DEFAULT FALSE,
  meal3_skipped   BOOLEAN DEFAULT FALSE,
  reminders_sent  INT DEFAULT 0,
  UNIQUE(user_id, date)
);
```

Поля `_skipped` означают — пользователь явно подтвердил что пропустил приём. Это не то же самое что не внёс данные — это осознанный выбор, бот фиксирует но не паникует повторно.

### 16.7 Тон паники — правила

1. **Никогда не осуждать.** "Бот не знает что ты ел" — не "ты облажался".
2. **Всегда давать лёгкий выход.** "Приблизительно" всегда лучше чем ничего — бот принимает любые данные.
3. **Конкретная причина.** Каждое напоминание объясняет ЗАЧЕМ это нужно — не просто "запиши!".
4. **Эскалация постепенная.** Первое напоминание мягкое, каждое следующее жёстче.
5. **Не спамить без смысла.** Если пользователь явно ответил "пропустил" — бот не шлёт ещё 3 напоминания по этому приёму.
6. **Паника — это забота, не наказание.** Тон: "мне важно знать как ты питаешься, потому что я слежу за твоим прогрессом".
